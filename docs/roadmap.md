# План реализации

## 1. Зафиксировать механику

Цель первой версии: ежедневная сетка 3x3. У каждой строки и каждого столбца есть условие, игрок вводит станцию в пересечении, ответ считается верным, если станция удовлетворяет обоим условиям.

Решения для MVP:

- размер сетки: 3x3;
- попытки: максимум 3 ошибки за партию;
- повтор станции: запрещен внутри одной сетки;
- транспортная область: сначала метро, МЦК и МЦД можно оставить в данных, но в настройках генератора нужно уметь выключать МЦД/будущие станции;
- режим: одна дневная загадка плюс кнопка новой тренировочной загадки.

## 2. Подготовить игровой датасет

Источник сейчас: `public/data/attributes.csv`.

Минимальный нормализованный объект станции:

- `id` - стабильный идентификатор строки станции;
- `groupId` - идентификатор станции как пересадочного узла;
- `nameRu`, `nameEn`;
- `lineId`, `lineNameRu`, `lineColor`;
- `tags` - список атрибутов из булевых колонок CSV;
- `searchText` - нормализованная строка для поиска.

Нужно отдельно решить, что игрок угадывает: станцию-линейный экземпляр или станцию-группу. Для MVP проще засчитывать строку станции, но в UX показывать линию, если у названия несколько вариантов.

## 3. Система признаков и подсказок

Категории подсказок должны быть:

- достаточно частотными, чтобы пересечения не были пустыми;
- понятными без энциклопедии;
- разнообразными: линии, эпохи, глубина, пересадки, тип платформы, буквы в названии, порядок на линии.

Стартовые типы условий:

- линия: `На Сокольнической линии`;
- эпоха: `Открыта до 1950 года`, `Открыта в 2010-е`;
- глубина: `Глубже 40 метров`, `Мелкого заложения`;
- пересадочность: `Есть пересадка`;
- название: `Название из одного слова`, `Начинается на гласную`;
- положение: `Конечная станция`, `Не конечная станция`;
- статус: `Не будущая станция`.

Для продакшена нужен отчет качества: сколько станций подходит под каждую подсказку и сколько вариантов есть в каждом пересечении.

## 4. Генератор загадок

Алгоритм MVP:

1. Собрать список кандидатов условий.
2. Случайно выбрать 3 условия для строк и 3 для столбцов.
3. Проверить все 9 пересечений.
4. Отклонить сетку, если есть пустые пересечения.
5. Отклонить сетку, если слишком много пересечений с одним и тем же очевидным ответом.
6. Сохранить seed, дату и список условий.

Улучшения:

- сложность по числу возможных ответов;
- запрет слишком похожих условий в одной сетке;
- ручной список ежедневных загадок;
- генерация нескольких кандидатов и выбор самой интересной.

## 5. Интерфейс

Первый экран должен сразу быть игрой, без лендинга. Основные элементы:

- заголовок и дата загадки;
- сетка 3x3 с условиями сверху и слева;
- поиск станции в активной ячейке;
- обратная связь: верно/неверно/уже использована;
- список найденных ответов и счет;
- кнопка новой тренировочной загадки.

Важно: не перегружать интерфейс справочным текстом. Подсказки должны быть самодостаточными.

## 6. Проверка и качество

Автоматические проверки:

- скрипт сборки данных завершается без ошибок;
- у каждой станции есть id, название и линия;
- каждая подсказка имеет минимум N подходящих станций;
- сгенерированная сетка имеет непустые пересечения;
- поиск находит станции с `ё/е`, дефисами и разными регистрами.

Ручная проверка:

- открыть локально через `python3 -m http.server 8000`;
- пройти одну сетку;
- проверить мобильную ширину;
- убедиться, что названия станций не ломают layout.

## 7. Релиз

Самый простой путь:

1. GitHub Pages для статического фронтенда.
2. Ежедневная загадка генерируется заранее и лежит в JSON.
3. CI обновляет `public/data/game-data.json` после изменения источника данных.
4. Позже можно добавить сервер/API только если понадобятся аккаунты, рейтинги или античит.

## 8. Открытые решения

- Название продукта: `Ветки в клетке` звучит живо, но можно проверить альтернативы.
- Включать ли МЦД в основной режим или сделать отдельный режим `вся рельсовая Москва`.
- Засчитывать ли пересадочный узел как один ответ или разные станции по линиям.
- Показывать ли редкость ответа, как в спортивных grid-играх.
- Нужны ли ежедневные архивы.
